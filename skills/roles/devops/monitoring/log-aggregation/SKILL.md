# 📝 Log Aggregation & Formatting

> **Use this skill when**: configuring how a service outputs logs to `stdout`, or setting up the central ingestion pipeline (ELK, Datadog) to parse those logs. Trigger: `/devops-config-logging`.
>
> **Out of scope**: This is not about writing the alert criteria (Use `alert-setup/SKILL.md`). This guarantees the log data is actually useful when queried.

---

## 🚫 Anti-Patterns

- **Raw Text Logging**: `console.log("User 123 failed to log in because bad password")`. A text string requires complex, brittle Regex to parse in the aggregation tool.
- **Log Splitting**: Outputting the same HTTP request across 4 separate log lines, causing them to be jumbled asynchronously in production.
- **Logging PII**: Emitting pure Customer Credit Card numbers or plain-text passwords into the logging stream, violating PCI-DSS compliance instantly.

---

## 🛠 Prerequisites & Tooling

1. A logging library adhering to structured JSON (e.g., Pino, Winston, Logrus).
2. A centralized log sink (Splunk, ElasticSearch, AWS CloudWatch).

---

## 🔄 Execution Workflow

### Step 1 — Enforce Structured JSON
Every log output destined for Production MUST be a JSON object securely printed to `stdout`.
```json
{
  "timestamp": "2026-03-30T10:15:30Z",
  "level": "error",
  "service_name": "auth-api",
  "trace_id": "abc-123",
  "message": "Authentication failed",
  "user_id": 405,
  "reason": "INVALID_PASSWORD_HASH"
}
```

### Step 2 — Trace ID Injection
Ensure a Correlation ID (`trace_id` / `request_id`) is generated at the system entry point (e.g., the API Gateway) and passed down to every downstream microservice.
Every single log line generated by that HTTP request must contain that exact `trace_id`.

### Step 3 — Define Log Levels Strictly
- `ERROR`: The system failed its job. Warrants investigation.
- `WARN`: Approaching a limit (e.g., API Rate Limit 90%) or using deprecated logic.
- `INFO`: State changes (Service started, Batch Job completed). DO NOT log every single 200 OK HTTP request here if traffic is high.
- `DEBUG`: Deep variable dumps. Must be disabled in Production to save Cloud costs.

### Step 4 — Log Scrubbing (Masking)
Implement a middleware scrubber *before* the log is emitted to `stdout`.
Scrub entirely: `password`, `token`, `secret`, `authorization_header`.

### Step 5 — Shippers & Aggregation
Never configure the App to write a `.log` file to disk on a container (it will be destroyed).
App prints to `stdout`.
A daemon (FluentBit, Promtail) running on the host captures `stdout`, batches it, and ships it to the central ElasticSearch/Datadog cluster.

## ⚠️ Error Handling (Fallback)

| Scenario | Condition | Fallback Action |
|----------|-----------|-----------------|
| Log Cost Explosion | Logging cluster bills $5,000 extra this month | Immediately flip the production environment log level from `INFO` to `WARN`. The backend is likely logging every generic health check ping. |
| Missing Stack Trace | The log says `ERROR` but has no context | Ensure the logging framework is capturing the exact `error.stack` parameter in a dedicated JSON field, not concatenating it onto the `message` string. |

---

## ✅ Done Criteria / Verification

Log Aggregation is functionally sound when:

- [ ] All production logs are emitted as strictly parsed JSON.
- [ ] A single `trace_id` can be searched in the SIEM tool to view the entire journey across 3 different microservices.
- [ ] Passwords and API keys do not exist in the centralized log search interface.
